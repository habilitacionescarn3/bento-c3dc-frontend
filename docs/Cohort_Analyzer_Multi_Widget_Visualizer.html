1. Project Overview

A planned overhaul of the Cohort Analyzer page to transform it into a customizable, grid-based dashboard where visualization cards (Venn diagrams, histograms, KM plots, tables) can be moved, resized, and organized into user-defined "tabs" or views.

Key Features:
  • Draggable and resizable visualization cards
  • Grid-based layout with snap points
  • User-configurable tabs/views
  • Layout persistence (localStorage)
  • Responsive design

Challenges Encountered & Solutions

Challenge 1: ESM Export Error (react-grid-layout v2.2.2)

Error: Can't reexport the named export 'DEFAULT_BREAKPOINTS' from non EcmaScript module

Cause: Webpack 4.28.3 has limited ES modules support. react-grid-layout v2.x uses ESM syntax incompatible with Webpack 4.

Solution: Downgrade to react-grid-layout@0.16.6 (same version used by cBioPortal).

Challenge 2: Optional Chaining Syntax Error

Error: Module parse failed: Unexpected token (270:24)
return this.props?.nodeRef?.current ?? _reactDom.default.findDOMNode(this);

Cause: Webpack 4 cannot parse ES2020 optional chaining (?.) or nullish coalescing (??) operators.

Affected packages: react-grid-layout v1.x/v2.x, react-draggable v4.x, react-resizable v2.x/v3.x

Solution: Pin to older versions without modern syntax.

Challenge 3: Transitive Dependency Issue

Error persisted after pinning versions:
./node_modules/react-resizable/node_modules/react-draggable/build/cjs/Draggable.js

Cause: react-resizable created a nested node_modules/react-draggable folder with a newer incompatible version.

Solution: Add npm overrides to force global version resolution.

Challenge 4: Infinite Loop on Layout Change

Error:
Maximum update depth exceeded. This can happen when a component repeatedly calls setState inside componentWillUpdate or componentDidUpdate.

Cause: The onLayoutChange callback fires on every render because fullLayout is recreated as a new array each render, and handleLayoutChange calls setLayout() unconditionally.

Solution: Compare layouts before updating state - only call setLayout() when the layout has actually changed.

5. Vision & Requirements

Grid Unit Concept

  • Minimum unit size: 100px
  • Dynamic calculation: units = Math.floor(containerWidth / 100)
  • Unit width: containerWidth / units

Example:

  • 700px container → 7 units × 100px each
  • 650px container → 6 units × ~108px each

Card Sizing

  • Cards defined in grid units: w: 7, h: 5 = 7 units wide, 5 units tall
  • Minimum constraints: minW, minH prevent cards from being too small
  • Cards snap to grid intersections when dragged/resized

Two-Tab Navigation (Core Feature)

  • Primary/Graphic View - Draggable/resizable tile grid with charts (Venn, KM plot, histograms, risk table)
  • Table View - Full-width participant data table (not draggable, fixed layout)
  • Default tab on load: Primary/Graphic View
  • State preserved when switching between tabs
  • Tab bar positioned at top of Cohort Analyzer (below header)

  1. VennDiagramCard - Interactive cohort intersection visualization
  2. HistogramCard - Configurable bar chart (Sex/Race/Treatment/etc.)

Tab Switching Logic

// CohortAnalyzerContext - Tab management
const [activeTab, setActiveTab] = useState('primary'); // 'primary' | 'table'

// Switch between tabs (preserves state)
const handleTabChange = (event, newTab) => {
  setActiveTab(newTab);
};

// Save Primary view layout changes to localStorage
const handleLayoutChange = (newLayout) => {
  setLayout(newLayout);
  localStorage.setItem('cohort-primary-layout', JSON.stringify(newLayout));
};

// Load saved layout on mount
useEffect(() => {
  const saved = localStorage.getItem('cohort-primary-layout');
  if (saved) setLayout(JSON.parse(saved));
}, []);

9. Approach Comparison: Hybrid vs Bespoke

Both approaches use react-grid-layout as the foundation. The question is whether to follow cBioPortal's patterns or design C3DC's own architecture.

Option A: Hybrid (cBioPortal-inspired)

What it means: Adopt cBioPortal's architectural patterns translated into C3DC's stack.

Advantages:

  • Proven architecture (years in production)
  • Reference implementation when stuck
  • Edge cases already solved
  • Faster initial development

Disadvantages:

  • Inherited complexity (18 chart types vs 5 needed)
  • Translation overhead (MobX → Context)
  • May not fit C3DC's tab-based UX
  • Harder to diverge later

Option B: Bespoke (C3DC's Own)

What it means: Design C3DC's grid architecture from first principles.

Advantages:

  • Tailored to C3DC's 5 chart types
  • Simpler, cleaner codebase
  • Matches tab-based UX vision
  • Easier to maintain
  • Material-UI native

Disadvantages:

  • More design decisions required
  • May hit unforeseen edge cases
  • Longer initial planning
  • No reference code

10. Final Recommendation

Recommended: Bespoke (C3DC's Own Design)

Estimated effort: ~12-14 story points

Rationale:

  1. Different UX paradigm - cBioPortal uses auto-layout; C3DC wants user-controlled tabs. Following their patterns would require significant modification anyway.
  2. Simpler requirements - 5 chart types vs 18. Building purpose-fit architecture avoids carrying unused complexity.
  3. Existing patterns - C3DC already has CohortAnalyzerContext. A bespoke solution can extend it naturally.
  4. Tabs feature is unique - cBioPortal doesn't have saved tabs/views. This core C3DC requirement means designing something new regardless.
  5. 100px unit spec - Your grid unit concept is different from cBioPortal's 195×170px fixed units.

When to Reconsider Hybrid:

  • If scope reduces significantly
  • If you discover complex edge cases during bespoke development
  • If cBioPortal adds a tabs feature you could leverage

Complexity/Cost Trade-off Analysis

Approach Complexity Story Points Technical Debt Risk
Hybrid Medium-High ~10 pts Higher (inherited patterns)
Bespoke (Recommended) Medium ~12-14 pts Lower (purpose-built)

Why the extra ~2-4 pts for bespoke is worthwhile:

  • Architecture tailored to C3DC requirements
  • Simpler, more maintainable code
  • Native tabs feature (not bolted on)
  • No inherited complexity from cBioPortal's 18-chart system
  • Lower long-term maintenance cost

Two additional tickets have been added that expand the scope and further clarify the tab system requirements.

New Ticket: Table View Tab

Acceptance Criteria:

  • "Table View" tab is visible next to Primary view
  • Tab navigation works correctly
  • Default landing remains Primary view
  • Table View loads without impacting performance

New Ticket: Tab-Based Navigation System

Description: Create a tab-based navigation system at the top of the Cohort Analyzer that allows users to switch between different views.

Scope:

  • Add a tab bar at the top of the Cohort Analyzer based on the design
  • Support switching between at least two tabs: Primary/Graphic view and Table View
  • Render each tab as a separate panel/view
  • Ensure only the active tab's content is visible
  • Preserve state when switching tabs
  • Maintain consistent styling with existing UI
